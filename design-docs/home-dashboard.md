## Purpose
A single **read-first** screen that answers, the moment you open it, **"where is every
project at?"** — without clicking into each one. Secondary: a **People** roster that
finally treats a recurring teammate as **one human across projects** (today "Tùng" in
project A and "Tùng" in project B are two unrelated rows), so you can see a person's
**combined load** across everything.

This un-defers the "global manage all projects" page that
[`project-member-settings.md`](./project-member-settings.md) explicitly parked
("considered and deferred in favor of current-project scope").

## User-facing behavior

### Navigation — the Home tile
- A **`Home` tile pinned at the very top of the icon rail** (above the project tiles),
  a muted `LayoutGrid`/`Home` lucide glyph in a squircle; active state ringed in accent
  exactly like a project tile (same `.tile-press` motion).
- Clicking it sets a **top-level `screen = 'home'`**: the sprint panel **and** main column
  are replaced by the full-width dashboard. Clicking any project tile — **or any project
  card on the dashboard** — sets `screen = 'project'` and lands you in that project exactly
  as today.
- **Does not replace the open-into-project habit.** The app still restores your last
  *screen* on reload (premise 1): if you were in a project, you come back to that project;
  if you were on Home, you come back to Home. Home is always one click away, never forced.

### The dashboard — projects grid (the hero)
A responsive grid of **project cards** (calm, card-per-group per `design-system.md`),
sorted by `createdAt` (same order as the rail). Each card is **read-only** and shows:
- **Identity** — project icon (`project.icon` else first letter) + name, tinted with
  `project.color ?? colorForName(project.name)`.
- **Active sprint** — reuse `sprintToSelect` (`lib.ts`), which is the **latest non-archived
  sprint by `startDate`** (it does **not** inspect today against the range — feed it sprints
  sorted by `startDate`). Shows its name + date range, or **"No active sprint"** when none.
- **Progress** — the active sprint's **leaf** tasks: `{pctDone}% · {remaining} left`
  (reuse the `CapacityBanner` leaf-task math — parents excluded). A **single-segment
  `pctDone`** bar (done/total), **not** the 3-way capacity stack — the card wants one number,
  not the done/in-flight/open breakdown.
- **Overdue flag** — a quiet `⚠ N overdue` chip (`--color-warn-ink`) when any non-done leaf
  task in the **whole project** (across all sprints + collections, archived included) is
  overdue; absent otherwise. **Overdue is whole-project on purpose** even though progress is
  active-sprint-scoped — the chip is a "something is slipping anywhere here" signal. Predicate
  (extract as a shared helper so the dashboard and `SprintView` agree — milestones are a
  `Task` with `estimate === 0`):
  - normal task: `dueDate != null && dueDate < today && status !== 'done'`
  - milestone (`estimate === 0`): `startDate < today && status !== 'done'` (its date lives on
    `startDate`, not `dueDate` — see `SprintView.tsx` milestone overdue rule).
- **People on it** — a small avatar stack of the project's members.
- Click anywhere on the card → enter that project.

### The dashboard — People roster (the support panel)
A **People** card on the **right** of the projects grid — confirmed layout (demo
`demo/home-dashboard-demo.html`, 2026-06-20). Fixed ~320px right column; **stacks below the
grid only as a responsive fallback** under ~920px (it does not become a primary layout
choice). One row per
**Person**, **sorted by combined load descending** (the overloaded surface first), each
showing:
- Avatar + name.
- **Combined load across all projects** — `{N} tasks · {effort}d`: count + summed
  `estimate ?? 0` (days; `null` unestimated counts as 0) over **all non-done leaf tasks**
  (across every sprint + collection) whose `assigneeId` is one of this person's member ids.
- **Project spread** — `in {K} projects` (small project dots/initials).
- **Next days off** — the soonest upcoming off-day, unioned across the person's members and
  deduped by date.

People is read-mostly, with the **minimum identity management** the feature needs (it can't
work without it — see below): rename a person, recolor, and **merge** two people into one.
The `⋯` manage popover is **portaled to `<body>`** and pinned to its trigger (with
viewport-edge flip/clamp) — the People panel uses `overflow-hidden` for its rounded corners,
so an in-panel `absolute` popover would be clipped (fixed 2026-06-20).

## Data model — Dexie **v13** (see [data-model.md](./data-model.md))

> **v12 is already in use** (member-lane-order `Member.order` backfill, `db.ts:420`). This
> feature is **v13**. The current latest store declaration is v10 (`db.ts:396`,
> `members: 'id, name, projectId'`).

### New table: `people`
```ts
Person { id: string; name: string; color: string; createdAt: number }
```

### `Member` gains `personId`
```ts
Member { id, projectId, name, color, daysOff, order?, personId?: string }
```
(`order?` already exists from the v12 lane-order migration — shown here so it's clear the
new migration must not disturb it.)

### v13 `.stores()` — re-declare the **full** index strings
Dexie `.stores()` **replaces** the schema for every table it names, so the new version must
restate the complete members index (append `personId`, keep `projectId`/`name`) and add
`people`. Carry the other tables forward unchanged from v10:
```ts
this.version(13).stores({
  people:  'id, name',                       // name indexed for dedupe/lookup
  members: 'id, name, projectId, personId',  // append personId — DO NOT drop projectId/name
  // projects/sprints/tasks/collections/events: unchanged — restate exactly as v10
}).upgrade(async (tx) => { /* backfill below — same version() */ })
```
Lets us query a person's members via `members.where('personId').equals(pid)`. The `people`
store **must be declared on the same `version(13)` whose `.upgrade()` writes into it** —
Dexie creates the object store before running that version's upgrade.

### Days off stays on `Member` — scheduler untouched (premise 4)
`Member.daysOff` is unchanged and the scheduler keeps reading it per project, exactly as
today. Person is **identity only** in v1. The dashboard's "next days off" is a pure
**display union** across a person's members (deduped by date). Moving days-off onto Person
is a deliberate **future** option, not this round (see Open questions).

### v13 upgrade callback (backfill)
Runs once on upgrade, all within the upgrade `tx` (read/write via the **`tx`-scoped**
tables, matching the v9 pattern at `db.ts:384` — `tx.table('members')`,
`tx.table('people')`):
1. Read all members; group by **normalized name** (`name.trim().toLowerCase()`),
   **globally across projects**.
2. For each distinct normalized name, `add` one `Person` (`name` = first member's display
   name, `color` = first member's `color`, which is always set, falling back to
   `colorForName(name)` defensively).
3. `update` every member's `personId` to its group's person id.

So recurring names auto-merge on upgrade. Mismatches ("Tùng" vs "Tung") are cleaned up
later via the manual **merge** action — they simply start as separate people.

### Linking on add — one write path (going forward)
**All member creation must route through a single `addMember(projectId, name)` helper** in
`db.ts`, because there are **two** existing `db.members.add` call sites
(`ProjectSettingsView.tsx:347` and the inline add in `SprintView.tsx:~2710`) — wrapping only
one would leave members with `personId === undefined` that vanish from the roster. The
helper: if a `Person` with the same normalized name exists, **link** to it; otherwise
**create** a new Person and link. Both call sites call `addMember()`.

### Project import
`remapBundle` / project import (`project-io`, used in `db.ts`) also creates members and
**bypasses both the one-time backfill and `addMember()`**. The import path must link each
imported member to a Person by normalized name (create if absent) as part of the remap, so
imported projects show up correctly in the roster.

## Implementation notes
- **`screen` state** in `App.tsx`: `'home' | 'project'`, persisted at
  `localStorage['plan-up:screen']`. `screen='home'` renders `<HomeDashboard/>` full-width
  (rail stays; sprint panel + main column hidden). The existing per-project selection
  (`currentProjectId`, `currentSprintId`, `selKind`, …) is **preserved untouched** so
  returning to a project restores it.
- **`HomeDashboard.tsx`** — all data via Dexie `useLiveQuery` (no stored aggregates):
  `projects`, all `sprints`, all `tasks`, all `members`, all `people`. Compute per-project
  active sprint + progress + overdue, and per-person load, in memory.
- **`people.ts`** — `personLoad(person, members, tasks)`, `personDaysOff(person, members)`,
  `mergePeople(srcId, dstId)`, `normalizeName()`. Pure where possible, unit-tested
  independently of React (mirrors `daysOffInRange`/`lib.ts`). **Merge semantics:** `dst`'s
  name + color are kept; every `src` member is reassigned `personId = dstId`; `src` is
  deleted. The UI picks the row you act on as `src` and the row you choose in "Merge into…"
  as `dst`.
- **Merge UX** — a person row → small popover: rename (`.editable`), recolor
  (`ColorSwatchRow`), **"Merge into…"** (pick the target person `dst`; confirm; reassigns
  this person's members into it).
- **Reuse:** `CapacityBanner` leaf math, `members.tsx` `Avatar`/`colorForName`/
  `ColorSwatchRow`, the settings-drawer popover pattern, `sprintToSelect` resolution.

## Rules & edge cases
- **Zero projects** → Home is the natural landing; shows "Create your first project"
  (wire the rail `+`). The app must not crash with `currentProject = null`.
- **Project with no sprint** → card shows "No active sprint", 0% progress.
- **Deleting a project / member** still happens only in the existing settings drawer
  (premise 2 — no second edit path). Deleting a member must **not** delete its Person.
  **Rule (decided):** a Person with zero remaining members is **kept and hidden** from the
  roster (so re-adding the name re-links to it) — not garbage-collected.
- **Days-off union** dedupes by date so the same off-day entered in two projects counts once.
- **`screen` persists** but Home is never auto-forced when a project screen was last active.

## Approaches considered
- **A — Minimal: projects-only dashboard, People deferred.** Home tile + grid; people only
  as per-project avatar stacks; no schema change. Rejected: under-delivers the confirmed
  cross-project people load.
- **B — Full dashboard + real People table (CHOSEN).** This document. One coherent ship;
  delivers both halves; scheduler and daily habit untouched; the only real risk is the v13
  migration/backfill.
- **C — Lateral: Home as the default front door.** Same data as B but Home becomes the
  launch screen (project-first opt-in). Rejected: fights the "land back where you were"
  rule for a tool used daily.

## Premises (agreed in session)
1. Home is a **new top-level destination**, not a replacement for opening into a project.
2. Project cards are **read-only**; all CRUD stays in the existing settings drawer.
3. The **same humans recur across projects** — confirmed; this is what justifies `people`.
4. Everything is **computed live** from existing data (no stored aggregates).
5. The `people` table (v13 migration + backfill + merge UI) is the **expensive 80%**; the
   project dashboard itself is cheap.

## Open questions
- **Person-level days off:** should `daysOff` eventually move from `Member` to `Person`
  (one person is off everywhere)? Deferred — would touch the scheduler. v1 keeps it on
  Member and unions for display only.
- **Empty Person GC:** keep zero-member people (for later re-link) or delete them? Default:
  keep + hide from roster.
- **Card progress scope:** active-sprint progress (chosen) vs whole-project progress.
  Revisit if the active-sprint number feels too narrow in use.

## Success criteria
- Open Home → see every project's active-sprint % done, remaining count, and overdue flag
  at a glance, and click into any project in one click.
- The People roster shows each recurring person **once**, with task/effort summed across
  all their projects, and lets you merge two rows that are the same human.
- v13 upgrade backfills cleanly on existing data; the scheduler produces identical dates
  before and after (days-off behavior unchanged). `npx tsc --noEmit && npm run build &&
  npx vitest run` all pass, including new `people.test.ts`.
