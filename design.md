# Design: ClickUp-style Sprint Planner (Self-hosted, No-auth)

Generated by /office-hours on 2026-06-03
Branch: main

Status: DRAFT
Mode: Builder

## Problem Statement

User muốn tự build lại web app plan/quản lý task giống ClickUp (sprint folder, group-by-assignee, biweekly sprint, monthly event folder) vì ClickUp tính phí per-seat đắt và team có nhiều người không cần quyền edit. Mục tiêu: giữ nguyên mental model ClickUp nhưng members là labels do user tự tạo (không cần login, không cần seat).

## What Makes This Cool

- **Bypass auth + billing complexity hoàn toàn.** Single-user tool nhưng vẫn assign được cho nhiều "ảo member". Không phải ai cũng cần multi-tenancy.
- **Giữ ClickUp mental model team đã quen** (Sprint Folder, Event Folder, Assignee group). Không phải học lại.
- **Own data của mình.** Lưu local, không phụ thuộc SaaS.

## Constraints

- Builder mode / side project — ship được, không cần production-grade.
- Wedge phải dùng được nhanh (1-3 buổi tối).
- Tech stack: React + Vite + IndexedDB (user chọn từ đầu, ready cho thêm view sau).
- Một user, nhiều "virtual members" (labels).

## Premises (agreed)

1. Single-user, no auth. Members = labels do user tự tạo.
2. Giữ ClickUp mental model: Sprint Folder (biweekly) + Event Folder (monthly) + Group-by-Assignee. KHÔNG redesign UX.
3. Wedge = 1 sprint view + CRUD task + assignee group + persistence.
4. Web app local-first. Sync/share là phase 2.

## Approaches Considered

### Approach A: Vanilla HTML + localStorage
1 file, Alpine.js, ship 1 tối. ✅ Nhanh. ❌ Khó scale sang multi-view.

### Approach B: React + Vite + Dexie/IndexedDB (CHOSEN)
Component-based, IndexedDB chứa data lớn, ready cho Board/Calendar view sau.

### Approach C: Markdown + Git sync
Task = .md file, sync qua GitHub. ✅ No lock-in. ❌ Conflict resolution + perf khi nhiều task.

## Recommended Approach: B

**Stack:**
- **Build:** Vite + React 18 + TypeScript
- **State/DB:** Dexie (IndexedDB wrapper) — schema versioning sẵn cho migrations
- **UI:** Tailwind CSS + shadcn/ui (table, dropdown, dialog) — match được look ClickUp
- **Table:** TanStack Table (group-by-assignee, sort, filter) — đây là core view, đầu tư tốt
- **Routing:** TanStack Router hoặc react-router (nếu cần multi-page sau)
- **Icons:** lucide-react

**Data model (wedge):**
```ts
Member { id, name, color, avatar? }
Sprint { id, name, startDate, endDate }
Task { id, title, assigneeId, sprintId, status, priority, dueDate, estimate? }
```

**Wedge scope (Tối 1-2):**
- Tạo/sửa/xóa Member (label + color). Xóa member → tasks fall back vào "Unassigned" group (không silent delete).
- Tạo/đổi Sprint (manual: tên, ngày bắt đầu, kết thúc) qua dropdown header + dialog.
- Tạo Task gán vào Sprint, gán Member, set status/priority/due. Inline edit toàn bộ.
- View chính: List của Sprint hiện tại, group by Assignee + Unassigned (replica view trong ảnh ClickUp).
- IndexedDB persistence.
- **Export / Import JSON** (bumped từ Phase 2 lên wedge — đây là backup duy nhất cho local-first app, clear browser data = mất hết, không thể defer).
- **Empty/error states:** IndexedDB unavailable (private mode) → message rõ ràng, không loading-vĩnh-viễn.
- Color member: deterministic từ `hash(name)` → cùng tên cùng màu, không random.

**Phase 2 (sau khi wedge dùng được):**
- Sprint Folder auto-create theo lịch biweekly. Spec: từ sprint hiện tại, tự sinh `Sprint N+1` với `startDate = currentSprint.endDate + 1`, `endDate = startDate + 13`. Tên format `Sprint {N}` parse từ tên cũ.
- Carry-over task chưa xong: button "Carry over" trên sprint cũ → tasks `status != 'done'` được set `sprintId = nextSprintId`. Confirm dialog liệt kê số task sẽ chuyển.
- Event Folder (monthly): khám phá thêm — có thể là Sprint dài 30 ngày với `kind: 'event'` flag, hoặc 1 bảng riêng. Defer cho đến khi user dùng wedge ≥1 tuần và biết mình thực sự cần.
- Capacity/effort banner (ai over/under) — cần `Task.estimate` được fill.
- Board view, Calendar view.
- Keyboard shortcut (j/k navigate, c create, e edit) — nếu speed là điểm phân biệt với ClickUp, đẩy lên đầu Phase 2.
- (Optional) Sync layer (Supabase/local file).

**Performance budget:**
- Target: 500 tasks / sprint vẫn render < 100ms. Vượt ngưỡng → memoize `tasksByMember`, virtualize TaskRow list.
- Inline edit title: debounce 300ms nếu thấy lag khi gõ nhanh.

**Schema migration:**
Dexie versioned schema. Khi đổi schema (vd: thêm `Task.labels: string[]`):
```ts
this.version(2).stores({ tasks: 'id, sprintId, assigneeId, status, createdAt, *labels' })
  .upgrade(tx => tx.table('tasks').toCollection().modify(t => { t.labels = [] }))
```
Mọi schema change phải có upgrade callback hoặc test fail.

## Open Questions

- **Sprint folder structure:** flat (Sprint 41, 42, 43) hay nested theo quý/năm? → defer, dùng flat trước.
- **Event folder:** chỉ là 1 sprint dài 1 tháng, hay schema khác? → defer.
- **Color/theme:** dark mode default hay light? → tùy taste, shadcn hỗ trợ cả hai.
- **Có muốn keyboard shortcut (j/k navigate, c create) ngay từ wedge không?** → quick win cho speed nếu user dùng nhiều.

## Success Criteria

- Mở web app local → tạo 5 task gán 3 member trong < 2 phút.
- Reload trang → data vẫn còn.
- Group-by-assignee view giống ảnh ClickUp gửi (tên người + tasks bên dưới + status badges).
- Xóa member → tasks không biến mất (chuyển vào Unassigned).
- Export JSON → Import JSON → data round-trip nguyên vẹn.
- IndexedDB blocked (private mode) → user thấy thông báo rõ, không loading vĩnh viễn.
- 0 task / 0 member → onboarding hint hiển thị (TODO: chưa làm, ghi vào Phase 2).
- User thực sự dùng nó cho sprint thật trong > 2 tuần.

## Distribution Plan

- Local dev: `vite dev` chạy trên localhost:5173.
- Deploy đơn: build static → host trên Vercel/Netlify/GitHub Pages (vẫn local-first, IndexedDB per-browser).
- Backup: tự implement Export JSON button, save xuống file. Import button để restore.
- Không CI/CD phức tạp — push GitHub + auto-deploy Vercel là đủ.

## Next Steps (concrete build tasks)

1. **Scaffold:** `npm create vite@latest plan-tmp -- --template react-ts`, cài tailwind, shadcn init.
2. **DB layer:** Cài Dexie, tạo `db.ts` với 3 table (members, sprints, tasks), seed 2-3 member mẫu.
3. **Members CRUD:** Settings page hoặc sidebar — list member, add/edit/delete, pick color.
4. **Sprint CRUD:** Sidebar list sprint, "+ New Sprint" với date picker.
5. **Task list + group-by-assignee:** TanStack Table, group by `assigneeId`, columns: title, due, priority, status. Inline edit status (dropdown), priority (flag).
6. **Add Task quick:** Inline "+ Add Task" row dưới mỗi assignee group (match ClickUp UX trong ảnh).
7. **Persistence verify:** reload — data còn.
8. **Backup:** Export/Import JSON button trong settings.

## What I noticed about how you think

- Mày correct mình giữa session ("tao nhầm, tao muốn giữ cái này") thay vì để tao build sai. Đó là feedback loop tốt — designers tốt làm thế.
- Mày không bị cám dỗ "full ClickUp clone" — chọn wedge nhỏ nhất khi tao đưa option. Hiếm.
- Mày chọn B (React + Vite) chứ không phải A (Vanilla) — signal là mày muốn cái này sống lâu hơn 1 tuần, hoặc mày đã quen React. Either way, đầu tư stack từ đầu hợp lý ở đây vì core view (group-by-assignee table) không dễ làm bằng vanilla.
- Pain point thật của mày là **economic** (ClickUp đắt per-seat) chứ không phải UX. Đó là một wedge rất rõ — "ClickUp without the seat tax" — và market lớn hơn mày nghĩ.
